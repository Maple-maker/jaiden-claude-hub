Section 01

The loop, exactly (not a metaphor)

When you start an agent, the SDK runs the same execution loop that powers Claude Code. It is a precise, documented cycle, not a vibe: Claude receives your prompt (with the system prompt, tool definitions, and history), evaluates and responds (text and/or tool calls), the SDK executes those tools and feeds the results back, and the cycle repeats. Each full cycle is one turn. The loop ends only when Claude produces a response with no tool calls — then you get the final result.

A quick question ("what files are here?") might be one or two turns. A real task ("refactor auth and update the tests") can chain dozens of tool calls across many turns, with Claude adjusting based on each result. Once you see runs as loops instead of single prompts, you stop trying to write the one perfect message and start designing the loop.

1  Receive prompt   (SystemMessage: init)
2  Evaluate + respond (AssistantMessage: text + tool calls)
3  Execute tools     (UserMessage: tool results)
4  Repeat 2-3        (each cycle = one turn)
5  Return result     (ResultMessage: text, cost, usage)
The reframeDon't ask 'what's the perfect prompt?' Ask 'what loop finishes this on its own — and where do I cap it?'
Section 02

A turn is the unit of work

A turn is one round trip: Claude outputs tool calls, the SDK runs them, and the results feed back automatically — without handing control back to you. Turns continue until Claude returns text with no tool calls. As the loop runs it streams typed messages so you can watch or drive it: a SystemMessage (init) with session metadata, an AssistantMessage per response, a UserMessage carrying each tool result, and finally a ResultMessage that ends the loop.

Anthropic's own walkthrough of "fix the failing tests in auth.ts" is four turns: turn 1 runs the tests (3 fail), turn 2 reads the source and test files, turn 3 edits and re-runs (all pass), and the final turn is a text-only summary. Three tool turns, one answer. That is the exact shape of a debugging session.

# Check the subtype before trusting output
async for message in query(prompt="Summarize this project"):
    if isinstance(message, AssistantMessage):
        print(f"Turn: {len(message.content)} blocks")
    if isinstance(message, ResultMessage):
        if message.subtype == "success":
            print(message.result)
Real use #1 — debugging in code reviewI let the loop run the multi-step debug (run tests, read, edit, re-run, report) and just set the guardrails. I review the diff, not every keystroke.
Section 03

Put a leash on it: turns, budget, effort

Without limits the loop runs until Claude finishes on its own. That is fine for well-scoped work and dangerous on open-ended prompts like "improve this codebase." Two caps stop runaways: max_turns (maximum tool-use round trips) and max_budget_usd (maximum spend). Hit either and the loop returns a ResultMessage with error_max_turns or error_max_budget_usd. A budget is the right default for any production or autonomous agent.

Separately, the effort option controls how hard Claude reasons per turn — low, medium, high, xhigh, max. Lower effort is cheaper and faster for simple jobs (listing files, a single grep); raise it for refactors and debugging. Effort is independent from extended thinking; you can mix them.

ClaudeAgentOptions(
    allowed_tools=["Read","Edit","Bash","Glob","Grep"],
    setting_sources=["project"],   # load CLAUDE.md, skills, hooks
    max_turns=30,                  # prevent runaway sessions
    max_budget_usd=2.00,           # hard cost cap
    effort="high",                 # thorough reasoning
)
How I set itDebugging loops get a turn cap so they can't spiral. I only raise effort to high/max for genuinely hard refactors — everything routine stays low to save cost.
Section 04

Why long loops get dumb — and the fix

The context window does not reset between turns. Everything accumulates: system prompt, tool definitions, conversation history, and every tool input and output. Large outputs (reading a big file, a verbose command) can burn thousands of tokens in a single turn, so long sessions carry far more weight than short ones.

When context approaches the limit, the SDK automatically compacts — it summarizes older history to free space and emits a system message with subtype compact_boundary. Compaction keeps recent exchanges but can drop specific instructions from early in the conversation. The fix: put persistent rules in CLAUDE.md (loaded via setting sources and re-injected on every request) instead of burying them in the first prompt, and offload big subtasks to subagents, which run in their own fresh context and return only a summary.

# CLAUDE.md — survives compaction, re-sent every request
# Summary instructions
When summarizing this conversation, always preserve:
- The current objective and acceptance criteria
- File paths read or modified
- Test results and error messages
- Decisions made and the reasoning
Real use #2 — reformatting the whole siteUpdating all 29 pages to the new format blew context fast. I moved the format spec into CLAUDE.md so every loop kept the rules even after compaction — then let it walk the repo file by file.
Section 05

Let it act — safely

Claude decides which tools to call; you decide what is allowed to run. Three options work together: allowed_tools auto-approves a list (a read-only agent with Read, Glob, Grep runs without prompting), disallowed_tools blocks tools regardless of other rules, and permission_mode governs everything not covered by an allow or deny rule. Useful modes: default (route to your approval callback), acceptEdits (auto-approve file edits and common filesystem commands), plan (explore without editing), and bypassPermissions (run everything — isolated environments only). When a tool is denied, Claude gets a rejection as the tool result and reroutes.

For speed, read-only tools (Read, Glob, Grep, read-only MCP tools) can run in parallel, while state-changing tools (Edit, Write, Bash) run sequentially to avoid conflicts. For safety and observability, hooks fire at fixed points — PreToolUse before a tool runs (validate or block), PostToolUse after it returns (audit, side effects), plus Stop, PreCompact, and subagent hooks. Hooks run in your process, not the agent's context window, so a PreToolUse hook that rejects a call short-circuits the loop without costing context.

# Block anything outside the repo, before it runs
def pre_tool_use(tool, args):
    if tool == "Bash" and "rm -rf" in args.get("command",""):
        return deny("blocked: destructive command")
    return allow()
My code-review guardrailacceptEdits so it can fix code autonomously, plus a PreToolUse hook that blocks any path outside the repo. Autonomy with a fence.
Section 06

Know when it stopped — and resume

The loop ends with a ResultMessage, and its subtype is the primary signal: success (finished normally — only this variant carries the result text), error_max_turns, error_max_budget_usd, error_during_execution, or error_max_structured_output_retries. Always check the subtype before reading result. Every subtype still carries total_cost_usd, usage, num_turns, and session_id, so you can track cost even on failures.

Capture session_id and you can resume later with the full context restored — every file read, every action taken. You can also fork a session to try a different approach without touching the original. This is what makes turn caps safe: if a long debug hits the limit, you resume instead of starting over.

if message.subtype == "success":
    print(message.result)
elif message.subtype == "error_max_turns":
    # resume with full context, no re-explaining
    resume(message.session_id)
else:
    print(f"Stopped: {message.subtype}")
The payoffCaps + resume = autonomous but recoverable. The loop can run long and you never lose the work.
